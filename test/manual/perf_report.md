# EffectiveSink 性能报告

## 测试环境

| 项目     | 值                                                                                |
| -------- | --------------------------------------------------------------------------------- |
| CPU      | AMD Ryzen 7 PRO 4750U (16 线程)                                                   |
| 编译器   | GCC 12.3.0, C++20, -march=native                                                  |
| Pipeline | EffectiveFormatter (protobuf) → Zstd 压缩 → AES-128-CBC 加密 → mmap 双缓存 → 落盘 |
| 缓存策略 | 双缓存 (master/slave)，Ratio > 0.8 触发 swap + 异步落盘                           |
| 构建类型 | **Release**                                                                       |

## 测试方法

每轮测试使用独立的 `EffectiveSink` 实例和临时目录，测试完成后 `Flush() + CleanDir()`，确保各轮互不干扰。每个场景先 warmup (200 次或 50/线程)，再计时。延迟测量单次 `Log()` 调用的墙钟时间。

## 1. 单线程吞吐

| 消息类型       | ops  | 耗时   | ops/sec     | avg  | p50  | p99  | max  |
| -------------- | ---- | ------ | ----------- | ---- | ---- | ---- | ---- |
| small (~50B)   | 3000 | 0.024s | **127,654** | 6us  | 4us  | 17us | 68us |
| medium (~200B) | 3000 | 0.034s | **88,461**  | 10us | 9us  | 19us | 75us |
| large (~380B)  | 2000 | 0.026s | **76,109**  | 11us | 11us | 20us | 80us |

```
small   |████████████████████████████████████████████████████████████████████████████████| 127k/s
medium  |████████████████████████████████████████████████████| 88k/s
large   |████████████████████████████████████████████| 76k/s
```

- **50B 短日志 12.7 万条/秒，单次延迟 6us**
- 消息体从 50B → 380B，延迟从 6us → 11us，退化平缓（protobuf + 压缩 + 加密的 O(N) 特征）
- **p50 ~ p99 差距极小**（4us → 17us），延迟分布极稳定，无异常抖动

## 2. 多线程并发（8000 ops, medium ~200B）

| 线程 | ops  | ops/sec    | avg   | p50  | p99     | max    |
| ---- | ---- | ---------- | ----- | ---- | ------- | ------ |
| 1    | 8000 | **88,246** | 10us  | 9us  | 20us    | 88us   |
| 2    | 8000 | **64,857** | 27us  | 10us | 184us   | 12.7ms |
| 4    | 8000 | **36,903** | 95us  | 25us | 1,212us | 27.2ms |
| 8    | 8000 | **39,548** | 164us | 25us | 4,946us | 30.6ms |

```
吞吐                    尾部延迟 (p99)
1 thr:  88k  ← best          20us
2 thr:  65k  ↓ 26%          184us   ↑ 9x
4 thr:  37k  ↓ 58%         1212us   ↑ 60x
8 thr:  40k  ↓ 55%         4946us   ↑ 247x
```

**瓶颈分析**：`Log()` 内所有操作（格式化 → 压缩 → 加密 → 写缓存）都在 `std::mutex` 保护下执行。多线程时锁竞争主导开销，吞吐下降、长尾延迟飙升。2 线程时 p50 仍与单线程持平 (10us)，说明轻竞争下中位延迟未受影响。

## 3. Flush 延迟

| 缓存条目 | Flush 耗时     |
| -------- | -------------- |
| 100      | 242us (0.24ms) |
| 1,000    | 444us (0.44ms) |
| 5,000    | 739us (0.74ms) |

- Flush 为 `write()` 系统调用，极快

## 6. 结论

**单线程性能已达生产级**——12.7 万条/秒、6us 延迟，P99 在 20us 以内，对绝大多数日志场景是过剩的。Pipeline 各环节架构清晰（protobuf → 压缩 → 加密 → mmap → 文件），模块可替换。

**多线程仍是当前主要短板**。`Log()` 一把大锁导致 8 线程吞吐仅为单线程的 45%。建议改进方向：
- 每个线程独立 `EffectiveSink` 实例（最简单，隔离性最好）
- 将压缩/加密移出临界区（`compress_` / `crypt_` 本身无状态，可无锁并发调用）
- 或使用无锁队列 + 单消费者线程异步写入
