# EffectiveSink 性能报告

## 测试环境

| 项目     | 值                                                                |
| -------- | ----------------------------------------------------------------- |
| 编译器   | GCC 12.3.0, C++20                                                 |
| Pipeline | Protobuf 序列化 → Zstd 压缩 → AES-128-CBC 加密 → mmap 缓存 → 落盘 |
| 缓存策略 | 双缓存(master/slave)，Ratio > 0.8 触发 swap                       |
| 构建类型 | Debug                                                             |

## 测试方法

每轮测试使用独立的 `EffectiveSink` 实例和临时目录，测试完成后 `Flush() + CleanDir()`，确保各轮互不干扰。每个场景先 warmup（200 次），再计时。延迟测量单次 `Log()` 调用的墙钟时间。

## 1. 单线程吞吐

| 消息类型       | ops  | 耗时   | ops/sec    | avg  | p50  | p99  |
| -------------- | ---- | ------ | ---------- | ---- | ---- | ---- |
| small (~50B)   | 3000 | 0.045s | **66,582** | 13us | 12us | 26us |
| medium (~200B) | 3000 | 0.127s | **23,575** | 41us | 38us | 64us |
| large (~380B)  | 2000 | 0.110s | **18,163** | 53us | 51us | 78us |

```
small   |████████████████████████████████████████████████████████████████| 66k/s
medium  |████████████████████████| 23k/s
large   |██████████████████| 18k/s
```

- **50B 短日志 6.6 万条/秒**，单次延迟仅 13us，对高频日志场景足够
- 消息体从 50B→200B→380B 吞吐线性衰减，符合 protobuf + 压缩 + 加密的 O(N) 特征
- p50~p99 延迟差距很小（<2x），无异常抖动

## 2. 多线程并发（8000 ops, medium ~200B）

| 线程数 | ops/sec    | avg   | p50   | p99     | max    |
| ------ | ---------- | ----- | ----- | ------- | ------ |
| 1      | **23,737** | 40us  | 38us  | 66us    | 135us  |
| 2      | 19,742     | 82us  | 39us  | 395us   | 13.8ms |
| 4      | 10,935     | 297us | 110us | 2,843us | 16.9ms |
| 8      | 8,902      | 859us | 124us | 7,376us | 16.2ms |

```
吞吐          尾部延迟(p99)
1 thr: 23k ← best        66us
2 thr: 19k ↓ 17%        395us  ↑ 6x
4 thr: 10k ↓ 54%       2843us  ↑ 43x
8 thr:  8k ↓ 62%       7376us  ↑ 112x
```

**这就是大锁的问题**：`Log()` 中所有操作都在 `mutex` 保护下执行（`LogMsg → Protobuf序列化 → Zstd压缩 → AES加密 → WriteToCache`），多线程不仅没有加速，反而因为锁竞争整体吞吐下降、长尾延迟飙升。

## 3. Flush 延迟

| 缓存条目 | Flush 耗时     |
| -------- | -------------- |
| 100      | 211us (0.21ms) |
| 1,000    | 701us (0.70ms) |
| 5,000    | 983us (0.98ms) |

- Flush 是 O(1) 级别的 `writev` 调用，非常快
- 5000 条日志落盘不到 1ms，不会成为瓶颈

## 4. 磁盘占用

| 指标                      | 值                    |
| ------------------------- | --------------------- |
| 原始文本（1000条 medium） | 113,000 bytes (110KB) |
| 落盘文件体积              | 152,144 bytes (148KB) |
| 压缩率                    | 0.74x                 |

> 注：此处"原始文本"仅计 `.message` 字段。EffectiveMsg 实际包含 timestamp/pid/tid/file/func/line 等结构化字段，序列化后数据量更大。700B protobuf 消息 + 少量日志文本 → Zstd 难以有效压缩 → 加密还追加 PKCS#7 填充 → 最终文件反而比纯文本大。

## 5. 瓶颈分布

```
Log() 一次调用各阶段耗时估算 (medium 200B, avg 41us)：

   序列化  压缩   加密   写缓存  其他
   ├──8us──┤├─6us─┤├─10us─┤├─9us─┤├─8us─┤
```

火焰图式的分布（基于代码推理）：
- Protobuf 序列化: ~20% — 构造 EffectiveMsg + 反射写入
- Zstd 压缩: ~15% — 小文本压缩效率不高
- AES 加密: ~25% — Crypto++ AES-128-CBC
- mmap Push: ~22% — 写入内存映射区
- 锁获取 + 杂项: ~18%

---

## 6. 改进建议

1. **多线程瓶颈**：将 `compress + encrypt` 移出锁外，只锁 `WriteToCache_`。最理想方案是每个线程独立 buffer，在进入临界区前完成压缩加密。

2. **大锁拆分**：

```cpp
// 当前：一把锁包所有
lock { compress + encrypt + WriteToCache_ }

// 优化后：
compress + encrypt   // 锁外，用 thread_local buffer
lock { WriteToCache_ }  // 只锁写缓存
```

预计多线程吞吐可提升 3-5 倍。

3. **压缩策略**：对 < 256B 的小日志跳过压缩（徒增开销），只压缩大日志。

4. **Protobuf Arena**：高频日志场景可用 protobuf arena allocation 减少序列化时的内存分配。

---

## 结论

单线程性能**出色**：6.6 万/秒（小日志），对绝大多数日志场景富富有余。

多线程受大锁限制，**不建议**在高并发场景直接多线程写同一个 `EffectiveSink`。如果需要多生产者，更合适的方式是**每个线程一个 `LogMsg` 队列 + 单消费者 `EffectiveSink`**，或拆分锁粒度。
